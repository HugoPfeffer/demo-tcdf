Prerequisites & Assumptions
Target Namespace: We will use zabbix-system for all deployments.

Cert-Manager: You have an existing ClusterIssuer (e.g., letsencrypt-prod or a local CA) already configured on the cluster.

VPC Topology: The OpenShift cluster and the RHEL EC2 instance reside in the same AWS VPC (or peered VPCs) to allow routing to the internal NLB.

Phase 1: Zabbix Server & Database (All-in-One)
We will leverage the Zabbix Operator to spin up the entire backend. The Operator will handle provisioning the PostgreSQL pod using AWS EBS (gp3-csi).

1. Operator Installation:
Install the Zabbix Operator via OperatorHub in the OpenShift Console into the zabbix-system namespace.

2. The ZabbixServer CR Manifest:
Once the Operator is running, deploy this Custom Resource. The Operator will read this and automatically spin up the Server, the Web UI, and a PostgreSQL database.

YAML
apiVersion: zabbix.com/v1alpha1
kind: ZabbixServer
metadata:
  name: zabbix-demo
  namespace: zabbix-system
spec:
  replicas: 1
  database:
    type: postgresql
    # The Operator will automatically provision a PVC for this DB
    storage:
      className: gp3-csi
      size: 10Gi
  web:
    replicas: 1
    service:
      type: ClusterIP

Phase 2: Exposing the Web UI (Native Route + Cert-Manager)
Since we want to show off OpenShift's native Route object while still automating the certificate via Cert-Manager, we will use Cert-Manager's native OpenShift Route support.

Instead of manually creating a Certificate CR and binding a Secret, the Red Hat Cert-Manager Operator allows you to simply annotate the Route. Cert-Manager will see the annotation, generate the cert, and dynamically inject it into the Route's spec.tls block.

1. The Annotated Route Manifest:

YAML
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: zabbix-web-secure
  namespace: zabbix-system
  annotations:
    # Triggers Cert-Manager to issue the cert
    cert-manager.io/cluster-issuer: "your-cluster-issuer-name" 
spec:
  host: zabbix.apps.your-cluster.domain.com
  to:
    kind: Service
    # Ensure this matches the Service name generated by the Zabbix Operator
    name: zabbix-demo-web 
  port:
    targetPort: 8080
  tls:
    # Edge termination allows OpenShift Router to handle the TLS offloading
    termination: edge
    insecureEdgeTerminationPolicy: Redirect

Phase 3: Exposing Port 10051 (Internal AWS NLB)
To allow the external RHEL EC2 instance to send agent data securely, we need to expose the Zabbix Server's raw TCP port (10051) using an Internal AWS Network Load Balancer.

1. The LoadBalancer Service Manifest:
We will create a supplementary K8s Service that targets the Zabbix Server pods specifically for agent traffic.

YAML
apiVersion: v1
kind: Service
metadata:
  name: zabbix-agent-trapper-nlb
  namespace: zabbix-system
  annotations:
    # Provisions a Network Load Balancer (NLB) instead of a Classic LB
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
    # Ensures the NLB is internal to the VPC (no public IP)
    service.beta.kubernetes.io/aws-load-balancer-internal: "true"
spec:
  type: LoadBalancer
  ports:
    - name: zabbix-trapper
      port: 10051
      targetPort: 10051
      protocol: TCP
  selector:
    # Ensure these labels match the Zabbix Server pod labels generated by the Operator
    app.kubernetes.io/name: zabbix-server
    zabbix-cluster: zabbix-demo
2. RHEL EC2 Agent Setup:
On your external RHEL EC2 instance, you will install the agent and point it to the internal DNS name of the AWS NLB generated by the Service above.

Bash
# Install the agent on RHEL 9
dnf install -y https://repo.zabbix.com/zabbix/7.0/rhel/9/x86_64/zabbix-release-7.0-1.el9.noarch.rpm
dnf install -y zabbix-agent

# Edit /etc/zabbix/zabbix_agentd.conf and update the Server parameter
# Server=<AWS_INTERNAL_NLB_DNS_NAME>

systemctl enable --now zabbix-agent
